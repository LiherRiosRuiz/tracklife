### Hallazgo de seguridad y resolucion

**Problema detectado:** `UserProfileController@show` registrada en `routes/api.php` FUERA del middleware `auth:sanctum`.
Endpoint era públicamente accesible: cualquier usuario sin token podía listar perfiles.

**Incoherencia anotada:** `UserSearchController@search` (operacional desde P3.3) sí exigía auth:sanctum.
Impacto: asimetria de seguridad — busqueda gateada, perfil no.

**Decision tomada por LIHER:** Mover ruta DENTRO del grupo `auth:sanctum`. Evidencia: consumidor siempre autenticado (`comunidad/perfil/[id]` usa `useApiData` con token); busqueda ya gateada (inconsistencia obvia).

**Resultado:** Ruta ahora coherente y protegida. Tests verifican 401 sin auth.

### Sesion LIHER: Bug crítico de persistencia — tests borraban usuarios de producción

**Sintoma reportado por el usuario:** "Tengo que crearme una cuenta nueva cada vez que quiero entrar; el login da Credenciales incorrectas."

**Investigacion (metodo empirico):** Se verificó que TODO el backend y el flujo de auth funcionaban de extremo a extremo — registro persiste en Mongo, sobrevive a reinicios del contenedor, token Sanctum sigue válido tras rotación de APP_KEY, `/api/auth/me` cross-origin OK (CORS correcto), BFF de Next (puerto 3000) persiste y setea cookie httpOnly. Único hecho objetivo: 0 usuarios en Mongo al inicio. Hipótesis descartadas una a una con evidencia: wipe de volumen (estable desde 4-jun, `make down` sin `-v`), doble hasheo de password (cuentas frescas loguean), cookie `secure` sobre HTTP (dev usa `secure:false`), confusión usuario/email en el form (envía email correcto).

**Causa raiz (probada):** Los tests de Laravel corrían contra la BD de PRODUCCIÓN `tracklife`, no contra una BD aislada. Las variables de entorno reales del contenedor Docker (`APP_ENV: local`, `MONGODB_URI: ...@mongodb:27017/tracklife`) definidas en `docker-compose.yml` ENSOMBRECÍAN por completo el `phpunit.xml` — ni `APP_ENV=testing` ni `MONGODB_URI` con `force="true"` se aplicaban. Verificado con test diagnóstico: en runtime de test `app()->environment()` = `local`, db resuelta = `tracklife`. El `tearDown()` de `MongoTestCleanup` hacía `drop()` de la colección `users` REAL en cada `php artisan test`. Como el propio flujo de re-enganche del roadmap indica "correr php artisan test", cada sesión de trabajo borraba los usuarios. El mensaje "Credenciales incorrectas" es ambiguo (login devuelve lo mismo si la contraseña falla o si el email no existe), disfrazando la pérdida de cuenta como error de contraseña.

**Prueba irrefutable:** usuarios producción 2 → correr `AuthTest` → 0. Y `tracklife_testing` nunca se creaba.

**Fix (commit `eca9b52`, dos capas):**
- `tests/TestCase.php`: `setUp()` fuerza en runtime la dsn mongodb añadiendo sufijo `_testing` al nombre de BD + `DB::purge('mongodb')`. Bypassea la resolución de env (que es justo lo roto). Idempotente.
- `tests/Traits/MongoTestCleanup.php`: guardia ALLOWLIST — lanza `RuntimeException` si la BD no termina en `_testing`. Red de seguridad: imposible volver a dropear producción.
- `tests/Feature/TestDatabaseIsolationTest.php`: test de regresión que fija el invariante (db name acaba en `_testing`).
- Verificado probando el negativo: sentinela en `tracklife.users` SOBREVIVE a la suite completa. 85 tests verdes contra `tracklife_testing`.

**Fix secundario (higiene):** `docker-entrypoint.sh` del api regeneraba la APP_KEY en cada arranque (rotándola, rompiendo estado cifrado). Ahora solo se genera si no existe. NO era la causa del síntoma, pero es un footgun real corregido.

**Leccion:** Nunca dejar que los tests apunten a la BD de producción. Las variables de entorno reales de Docker (`environment:` en docker-compose) pueden ensombrecer `phpunit.xml`, incluso con `force="true"`. La defensa fiable es forzar la BD de test en el `TestCase` base + una guardia allowlist en el cleanup, no confiar en la resolución de env.
