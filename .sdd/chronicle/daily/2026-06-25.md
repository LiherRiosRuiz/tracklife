### Hito 3: P3.2 BLOQUEADO — Veredicto de Platon + opciones de arquitectura auth

**Sprint P3.2 objetivo:** "Server Components en Dashboard"

**Analisis de Platon:** NO VIABLE en la forma planteada. Razon:

**Problema de arquitectura:**
- Token Sanctum se persiste en `localStorage` (client-only)
- Server Components (Next.js) no pueden acceder a localStorage
- Las 19+ paginas que usan `token!` del AuthContext no pueden fetchar
  datos autenticados server-side sin refactor de auth previo

**Opciones identificadas:**

| Opcion | Descripcion | Viabilidad | Notas |
|--------|-------------|-----------|-------|
| A (recomendado) | **BFF incremental** — Backend Node.js como proxy de autenticacion. Cookie httpOnly para token, dual-write `localStorage` (para compatibilidad client durante migracion). Permite Server Components graduales sin romper auth | VIABLE | Refactor progresivo, no afecta API Laravel, mas seguro (httpOnly XSS-resistant) |
| B (descartado) | Cookie no-httpOnly | NO | Misma exposicion XSS que localStorage, sin beneficio |
| C (norte LP) | Sanctum cookie-SPA completo | VIABLE LP | Toca backend/CORS/session config. Incompatible con SPAs actuales. Para refactor mayor en P5+ |

**Decision pendiente del usuario:**
- Opcion A (refactor BFF incremental) → desbloquea P3.2 en 1-2 sesiones
- Mantener en `docs/Pendientes.md` como **BLOQUEADO** hasta decision
- Anotar opciones A/C en `docs/Roadmap TrackLife.md`

**Estado del roadmap:**
- P3.2 marcado como **BLOQUEADO** pendiente refactor auth (decision usuario)
- P3.3 (busqueda usuarios) no tiene dependencias — puede adelantarse si se prefiere
- Proxima sesion: aclarar que prioridad (P3.2 con BFF refactor vs. P3.3 feature primera)

**Push a GitHub:** package-lock.json cambios de npm install anadidos; commit pendiente.
